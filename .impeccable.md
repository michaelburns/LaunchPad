# LaunchPad — Design Context

## Users

LaunchPad is a single-host operator console for IT staff to launch, schedule, and audit PowerShell scripts. The current target is **showcase/portfolio** — peers and recruiters open it as a "wow, this is a *good* internal tool" reference — but the actual users are sysadmins, DevOps, and ops generalists who fire scripts during the workday and need to see what's running, what failed, and what's queued.

Job-to-be-done: pick a script, supply parameters, fire it (or schedule it), watch it run, read the output. Secondary: audit who ran what when, manage roles & categories.

## Brand Personality

**Operator-flavored. Dense. Quietly opinionated.**

The reference set is **Railway, Fly.io, Linear, Tailscale admin, Vercel dashboard** — modern engineering dashboards that have *taste* but still look like tools. They earn personality through information density, mono-prominent typography, and one or two signature moves — not by dressing up as a different category of product.

The previous pass (April 2026) overcommitted to an editorial/magazine metaphor — mastheads, "Edition IX," "filed by," colophons. That was wrong. LaunchPad is a tool. It should look like a tool that someone with taste built, not like a tool pretending to be a publication.

## Aesthetic Direction

- **Theme**: Dark by default. Operators run scripts during long debugging sessions; the output console is already dark; the chrome should match. Build elevation through lighter surfaces (per dark-mode rules), not shadow.
- **Palette**: Deep blue-black surfaces with three elevation steps. **One signal-orange accent (sodium-amber, ~oklch(72% 0.16 60))** — used for the active moment (running job, focused field, primary action). Never purple, never cyan-on-dark, never neon green.
- **Type**:
  - **Recursive Sans** (variable, Google Fonts) — labels, headings, chrome. Use the linear axis for restraint, the heavier weights for emphasis.
  - **Recursive Mono Casual** (same family, MONO axis on, CASL axis up) — IDs, durations, status, command, output. The casual axis is the "slightly weird" — its lowercase `g`/`a` have a hand-drawn quality that one Google Fonts pairing won't.
  - One family, two visually distinct variants, total cohesion.
- **Layout**: **Top bar (48px)** with brand + context + user. **Left sidebar (~220px)** with nav and live counters. **Main content** fills the rest. Density beats whitespace — operators want to see info, not gardens.
- **Signature**: The script-execution view is the soul of the product. Chrome stays quiet so when you're staring at a running job — pulsing accent dot, live-streaming output on the dark surface, params echoed back — that view feels designed and the rest feels considered.

### Anti-references (explicit)

- **Editorial / magazine** — no mastheads, no roman numerals, no "Vol. IX," no "filed by," no colophon, no Bodoni, no "operator's gazette." LaunchPad is not a publication.
- **Bootstrap admin theme** — no fixed dark navbar with default cobalt-blue buttons, no card grids of stats with icon + label + number.
- **Cyberpunk hacker dashboard** — no cyan-on-black, no matrix rain, no glitch text, no neon green CRT glow.
- **Glassmorphism** — no blur backdrops, no glow borders, no frosted overlays.
- **Generic "AI SaaS dark"** — purple-to-blue gradients, glowing accent on dark, hero-metric template. The whole point is to look like a serious engineering tool, not a slide-deck render.

## Design Principles

1. **Chrome is quiet, content is loud.** The shell, nav, and headers exist to disappear behind the script roster, the running job, the output stream. If a chrome element draws the eye, it's wrong.
2. **Mono carries the meaning.** Status text, IDs, durations, timestamps, command names, output — all monospace. Sans only for labels and prose chrome. The mono/sans split *is* the visual hierarchy.
3. **One accent, used rarely.** Sodium-amber appears only on the *active* state — the running job, the focused input, the primary action of the moment. Everywhere else is ink-on-paper-but-inverted.
4. **Density without chaos.** Hairlines (1px) instead of borders (≥2px). Tabular numerals always. Tight line-heights. Information per screen is a feature.
5. **Execution view first.** Every chrome decision is judged by: "does this make the running-script view land harder?" If a chrome flourish competes with the live job, the flourish loses.
